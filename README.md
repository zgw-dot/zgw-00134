# 校餐过敏投诉复盘看板

### 项目简介

校餐过敏投诉复盘看板，帮助学校快速发现、归并、复核校餐过敏风险事件，解决多源数据（菜单/档案/领餐/投诉）分散难归集、风险滞后发现、复核留痕缺失等核心问题。

### 技术栈

- React 18 + TypeScript + Vite
- Zustand（状态管理 + localStorage 持久化）
- Tailwind CSS（样式）
- Lucide React（图标）
- 纯前端应用，数据全部存储在浏览器 localStorage

### 启动步骤

```bash
# 1. 安装依赖
npm install

# 2. 启动开发服务器
npm run dev
# 打开 http://localhost:5173

# 3. 生产构建
npm run build
```

### 数据文件准备

项目内置了样例数据位于 `sample-data/` 目录：

| 文件 | 内容 | 说明 |
|---|---|---|
| menu.json | 菜单 | 含2条坏数据（缺meal_date/空dish_id）用于测试校验拦截 |
| profiles.json | 学生过敏档案 | 5位学生，含6类过敏原，含 severe 级别 |
| pickups.json | 领餐记录 | 8条记录，关联菜单与学生 |
| complaints.json | 投诉记录 | 3条（1条缺student_id用于校验拦截） |
| alias-config.json | 过敏原别名映射 | 6类过敏原的别名表 |

### 核心功能

#### 1. 数据导入

- 顶部「导入数据」按钮打开弹窗，分别选择4类文件进行导入
- **自动校验**：
  - 菜单：拦截缺 meal_date 或 dish_id 为空的行
  - 投诉：拦截格式错误/缺失必填字段的行
  - 档案 / 领餐：检查字段完整性
- **重复导入保护**：基于文件内容哈希，相同文件不会被重复导入
- 导入后自动重新归并风险事件，**已有事件的复核日志不会被覆盖**（仅追加新证据）

#### 2. 风险事件归并

按以下维度自动归并风险事件：
- 过敏原标准名（通过别名表统一映射，如「花生酱」「花生油」统一归为「花生」）
- 餐次（早餐/午餐/晚餐/点心）
- 时间窗（±2小时窗口）

风险等级判定规则：
- **高风险 HIGH**：档案严重过敏 + 有领餐记录 + 收到投诉
- **中风险 MEDIUM**：档案过敏+领餐匹配，或档案+投诉，或单独有投诉
- **低风险 LOW**：仅原料标签匹配

#### 3. 看板筛选

筛选器支持多维度组合：
- 班级多选下拉
- 餐次胶囊按钮
- 风险等级（高/中/低）
- 事件状态（待处理/已确认/误报/已关闭）
- 全文搜索（学生名、过敏原、班级、备注）

#### 4. 事件流转与复核

点击事件卡片打开右侧详情抽屉，在「复核工作台」区域：
- 选择新状态：待处理 → 已确认 / 误报 / 已关闭
- 填写复核备注
- 点击保存后，流转日志会按时间线记录

#### 5. 证据溯源

每个事件包含 4 类来源证据，可展开手风琴查看详情并复制原始 JSON：
- 📋 菜单行：命中的过敏原对应的菜单菜品
- 👤 过敏档案：涉及学生的过敏信息
- 🍽️ 领餐记录：学生取餐的时间和菜品
- ⚠️ 投诉记录：收到的投诉和具体症状

#### 6. 数据导出

右上角「导出」菜单支持：
- **CSV 导出**：适合 Excel 打开的明细
- **JSON 导出**：适合二次处理

导出仅包含**当前筛选后可见的事件**，隐藏/被筛选排除的事件不会混入。
文件名含日期时间戳。

#### 7. 过敏原别名配置

顶部「过敏原别名」按钮打开配置弹窗：
- 左侧填写过敏原标准名
- 右侧填写别名（逗号、顿号分隔）
- 点击保存时自动校验循环引用和别名重复

### 三条验收链路（核心测试路径）

#### 链路一：主流程 - 样例数据导入 → 生成事件 → 复核 → 导出

**操作步骤**：
1. 启动后访问看板，顶部点击「导入数据」
2. 依次导入 `sample-data/menu.json` → 过敏档案 `profiles.json` → 领餐记录 `pickups.json` → 投诉 `complaints.json`
3. **观察**：导入结果显示 菜单5条通过+2条错误，投诉2条通过+1条错误（拦截成功）
4. 关闭弹窗 → **看板应显示约 5-8 个风险事件**，其中1个应为 HIGH 高风险（张小明 花生 severe + 午餐领餐 + 投诉）
5. 点击高风险事件卡片 → **右侧抽屉打开**：
   - 基本信息显示过敏原为「花生」、餐次「午餐」、含证据 3-4 类
   - 展开「菜单」「档案」「领餐」「投诉」证据链 → **能看到对应 source_data**
   - 在复核区域选「已确认」，备注「与投诉描述一致，确认为花生过敏」→ 保存
   - 流转日志出现一条新记录，顶部统计栏「已确认」+1，「待处理」-1
6. 点击顶部「导出」→ CSV → 打开文件 → **检查**：应包含所有可见事件；如果之前筛选了「高风险」，导出只含高风险行
7. 刷新浏览器（F5）→ **验证持久化**：事件仍在、状态仍为「已确认」、备注、关闭时间、流转日志全部保留

**验收标准**：✅ 所有步骤成功；✅ 导出CSV可正常打开；✅ 刷新数据不丢失。

---

#### 链路二：失败路径 - 拦截坏数据 + 重复导入 + 别名校验

**操作步骤**：
1. 新建测试文件 `bad-menu.json` 内容：`[{"dish_name":"test","meal_type":"午餐"},{"dish_id":"","meal_date":"2026-06-15"}]`
2. 导入此文件 → **观察**：2条全部被拦截，导入结果0条成功+2条错误（分别提示缺 meal_date 和 dish_id 为空）
3. 新建 `bad-complaint.json` 内容：`{"invalid": true}`（不是数组）→ **观察**：顶层必须是数组，导入失败
4. 再次导入 `sample-data/menu.json`（之前已经导入过）→ **观察**：提示「该文件已导入过，请勿重复导入」，事件数量没有变化
5. 打开「过敏原别名」配置：
   - 添加一行标准名「小麦」，别名「花生酱」→ **保存时提示别名重复**
   - 添加一行标准名「花生」，别名「花生」→ **提示标准名别名包含自身**
   - 添加一行空的标准名 → **保存时忽略空行**

**验收标准**：✅ 缺字段全部被拦截；✅ 非法JSON格式被拦截；✅ 重复导入被拒绝；✅ 别名配置错误全部能被检测并给出具体错误消息。

---

#### 链路三：筛选与状态流转 - 筛选导出不混入隐藏事件 + 复核保护

**操作步骤**：
1. 确保已导入链路一的全部样例数据，看板有多个事件
2. 筛选器点击「高风险」→ 事件列表只剩下高风险事件（1-2个）
3. 记录此时的可见事件数 N
4. 点击「导出」→ CSV/JSON → 打开导出文件数行数
5. **验证**：数据行数 = N（表头除外），非高风险事件**没有被混入**
6. 手动把所有事件都流转为「已关闭」并填写不同备注
7. 再次导入 `sample-data/profiles.json`（故意导入档案）
8. **验证**：
   - 所有事件的「已关闭」状态**没有被改回待处理**
   - 每条事件的 `review_logs` 数量不变，之前的备注仍然保留
   - `closed_at` 时间戳依旧存在
9. 筛选「已关闭」→ 全部事件仍可见，状态正确

**验收标准**：✅ 筛选后导出文件行数正确，不包含被隐藏事件；✅ 新导入数据不覆盖已有复核状态/备注/关闭时间/流转日志。

### 常见问题

- **Q: 数据存在哪里？** A: 所有原始数据、事件、状态都保存在浏览器 `localStorage`，键名 `allergy-board-store`。清除浏览器数据会删除所有数据，请在清空前先导出备份。
- **Q: 如何清空重置？** A: 浏览器控制台执行 `localStorage.removeItem('allergy-board-store')` 然后刷新页面。
- **Q: 为什么导入文件时部分行被跳过？** A: 点击导入结果中的「错误列表」展开，可查看每条被拦截的具体原因和字段值。
- **Q: 别名如何工作？** A: 所有命中的原料词会先查别名表，映射到统一的标准名后再进行归并。例如「虾子」「虾仁」都会归为「甲壳类」。
